# Constitution

Immutable rules for the TRACT Geolocation Formatter QGIS plugin. Every spec, plan, and
implementation must respect these constraints. Violations require an explicit constitutional
amendment (update this file with a rationale) before proceeding.

Domain-specific rules live in `domains/`. When a domain file conflicts with this constitution,
the domain file takes precedence for work within that domain's scope.

---

## 1. Architecture

- **Single-class plugin with dialog.** The plugin is a standard QGIS Python plugin generated by Plugin Builder. The entry point is `__init__.py` → `classFactory(iface)` → main class in `TRACT_Geolocation_Formatter.py`.
- **Main class** (`TractGeolocationFormatter`) lives in `tract_geolocation_formatter/TRACT_Geolocation_Formatter.py`. It owns the plugin lifecycle (`initGui`, `unload`, `run`), dialog management, geometry processing helpers, and the main transformation pipeline.
- **Dialog** is split into a wrapper (`TRACT_Geolocation_Formatter_dialog.py`) and a generated base (`TRACT_Geolocation_Formatter_dialog_base.py`). The base is auto-generated from the `.ui` file — do not edit the base file manually. UI changes go through the `.ui` file and regeneration via `pyuic5`.
- **Processing pipeline runs in-process** inside QGIS. There is no background thread — long operations use `QApplication.processEvents()` with a progress bar in the QGIS message bar.
- **All geometry logic lives in the main class** as private helper methods (`_truncate_geometry_coordinates`, `_fix_duplicate_vertices`, `_get_tract_geometry_errors`, etc.). If the class grows too large, extract geometry helpers into a dedicated module under `tract_geolocation_formatter/` — but keep the public API surface minimal.

## 2. Processing Pipeline

The transformation pipeline in `_run_transformation_from_dialog` follows a strict order for each feature:

1. **Skip** empty/null geometries
2. **Detect** Z values (log warning, Z is dropped at write time via 2D WKB type)
3. **Reproject** to EPSG:4326
4. **Truncate** coordinates to 6 decimal places (not round — truncate)
5. **Fix** consecutive duplicate vertices (truncation can create them)
6. **Validate** with `isGeosValid()` → `makeValid()` → extract polygonal parts
7. **TRACT geometry checks** (duplicate vertices in rings, <3 unique coords, boundary self-intersections via Shapely)
8. **Minimum area check** (transform to EPSG:6933 equal-area CRS, compare against threshold)
9. **Polygon holes detection** (interior rings)

This order matters — each step depends on the previous. Do not reorder without updating the full pipeline logic.

## 3. Output Contracts

- **GeoJSON output** must contain at minimum the four TRACT attributes: `NodeID` (String), `PlotID` (String), `TRACTStatus` (String), `TRACTIssue` (String). Original input fields are preserved alongside them. Fields mapped to NodeID/PlotID via "Use existing field" are excluded. Input fields that clash with TRACT field names are renamed with an `_orig` suffix. <!-- Amended 2026-04-14: DEV-0001 — preserve original fields in output -->
- **TRACTStatus** values: `"READY"` (no issues) or `"NEEDS_FIX"` (one or more issues detected). No other values are valid.
- **Coordinate precision:** all output coordinates are truncated to 6 decimal places. This matches the EU TRACES system format.
- **CRS:** output is always EPSG:4326, regardless of input CRS.
- **CSV validation report** is written alongside the GeoJSON. It contains one row per issue per feature, with columns: `feature_id`, `NodeID`, `PlotID`, `status`, `issue_type`, `message`.
- **Output layer is auto-loaded** into the QGIS project after export.

## 4. Dependencies

- **QGIS 3.x** (with its bundled Python) is the runtime. The plugin uses `qgis.core` and `qgis.PyQt` APIs exclusively — no direct PyQt5 imports except in the generated dialog base.
- **Shapely** is used for boundary self-intersection checks (`shapely.geometry.shape`). It is the only external geometry library. Do not introduce additional geometry libraries (e.g., geopandas, fiona) unless strictly necessary.
- **No network calls.** The plugin is fully offline. Do not add any HTTP requests, API calls, or telemetry.
- **No database.** All data comes from QGIS layers and is written to local files.

## 5. Constants

- **`MIN_PLOT_AREA_HA = 0.05`** — minimum polygon area in hectares.
- **`AREA_CRS = EPSG:6933`** — equal-area CRS used for area calculations.
- **`COORD_DECIMALS = 6`** — coordinate truncation precision.

Constants are defined at module level in the main plugin file. If more constants are needed, keep them in the same location unless a separate config module is justified.

## 6. Code Quality

- **Python version:** must match the Python bundled with QGIS 3.x (currently 3.12+).
- **No external linter/formatter is currently configured.** Follow PEP 8 conventions. Keep line length reasonable (~120 chars).
- **Commits:** use descriptive commit messages. No strict conventional commit format is enforced yet.
- **Tests:** `tests/` directory exists but is currently empty. New features should include tests where feasible (see `conventions/testing.md`).

## 7. Security

- **No secrets in code or specs.** No `.env` values, no API keys, no tokens.
- **No user data leaves the machine.** The plugin processes local files and writes local output. This is a hard constraint.

## Amending the Constitution

These rules are immutable by default. To change one:

1. **Create a decision record** in the active spec's `decision-log.md`:
   - What rule is being changed
   - Why it no longer serves the project
   - What the new rule is
   - What existing code is affected

2. **Update this file** with the new rule. Add a comment with the date and reason:
   ```markdown
   - **New rule.** <!-- Amended YYYY-MM-DD: reason -->
   ```

3. **Update affected convention files** in `conventions/` to stay consistent.

Amendments should be rare. If you're amending frequently, the rules may be too specific.
