# Constitution

Immutable rules for the Tract Traceability Admin API. Every spec, plan, and implementation
must respect these constraints. Violations require an explicit constitutional amendment
(update this file with a rationale) before proceeding.

Version-specific rules (e.g. v3 migration) live in `domains/`. When a domain file conflicts
with this constitution, the domain file takes precedence for work within that domain's scope.

---

## 1. Architecture

- **Layered architecture is the default.** Request flow: Router -> Controller -> Service -> Repository -> Database.
- **Controllers** live in `src/api/{version}/controllers/`. They handle HTTP concerns (status codes, request parsing, auth dependencies) and delegate business logic to services.
- **Services** live in `src/services/`. They contain business logic and orchestrate repository calls. New services MUST NOT import FastAPI HTTP types (`HTTPException`, `Request`, `Response`, `JSONResponse`) — see § 4 Layer Boundaries.
- **Repositories** live in `src/sqlalchemy_pg/repositories/`. New repositories MUST inherit from `BaseRepository[ModelT, FilterT]` (`src/sqlalchemy_pg/base/base_repository.py`). Legacy repositories may use custom patterns (static methods, raw queries) — do not extend these; migrate to `BaseRepository` when modifying them. BigQuery-specific query classes live in `src/sqlalchemy_bq/`.
- **Pydantic models** (request/response) live in `src/models/`. ORM models live in `src/sqlalchemy_pg/models/`.
- **Middleware integrations** (e.g. SAP) have their own `src/middlewares/integrations/` subtree with controllers, services, repositories, and models scoped to the integration.
- **v2 is the primary API version** for existing endpoints. v3 is used for migrated endpoints (see `domains/v3-migration.md`). New endpoints follow whichever version the domain specifies.

## 2. Database

- **PostgreSQL is the transactional store. BigQuery is the analytical store.** Know which one you are writing to.
- **Every PostgreSQL table must have an ORM model in this repo.** SQLAlchemy models are the single source of truth for schema.
- **Migrations** are generated with `alembic revision --autogenerate`. Do not write migration SQL by hand unless autogenerate cannot express the change (e.g., raw SQL data migration, custom trigger logic).
- **New models must inherit from `Base`** (not `LegacyBase`, not `DocumentBase`). `Base` provides `id` (UUID, `gen_random_uuid()`), `created_at` (TIMESTAMPTZ), `updated_at` (TIMESTAMPTZ). Do NOT redeclare `id` in subclasses.
- **`updated_at` triggers are auto-injected** by the `rewriter.Rewriter()` in `alembic/env.py`. It rewrites `CreateTableOp` and `AddColumnOp` to append a `CREATE TRIGGER` when the table has an `updated_at` column. Verify the trigger appears in autogenerated migrations before committing.
- **Use `server_default=func.current_timestamp()` in ORM models.** Autogenerate translates this to `sa.text('CURRENT_TIMESTAMP')` in migrations. Hand-written migration code must use `sa.text('CURRENT_TIMESTAMP')` to match.
- **All new resource identifiers MUST use UUIDv4.**

## 3. API Conventions

- **Response envelope:** `{ "data": ..., "meta": {...} }`.
- **Pagination:** `?page[limit]=N&page[offset]=N`
- **Filtering:** `?filter[fieldName]=value`
- **Sorting:** `?sort=field,-otherField`
- **Resource URLs:** plural, kebab-case, verb-free nouns only.
- **HTTP verbs:** GET (read), POST (create), PATCH (partial update), PUT (full replace), DELETE (remove).
- **All routes are protected by Keycloak JWT auth** except `/api/health`, docs, and pubsub endpoints. Use `RolesChecker` dependency for RBAC.
- **Roles:** DATA_VIEWER, DATA_UPLOADER, DATA_ADMIN, DATA_SUBMITTER. (`DATA_SUPPLIER` is deprecated — do not use in new code.) New roles require Keycloak infrastructure changes.

## 4. Layer Boundaries

- **Controllers** own the HTTP boundary and session lifecycle (commit on success, rollback on failure). They MUST NOT contain domain policy. They SHOULD delegate to services, but MAY call repositories directly for trivial passthrough reads with no business logic (e.g., get-by-id, list-all with filters only). If there is any validation, transformation, or orchestration — use a service.
- **Services** own business logic. They coordinate writes but MUST NOT commit — the controller owns the transaction boundary. They MUST NOT raise `HTTPException` or import FastAPI HTTP types; use domain exceptions instead. (Legacy violations exist in `data_request_service`, `signed_url_service`, `override_evidence_request_service` — do not propagate this pattern.)
- **Repositories** own data access. They flush after writes (`session.flush()`) but MUST NOT commit. (Legacy violations exist in `master_data_repository`, `supplier_node_repository`, `risk_document_repository` — do not propagate this pattern.)
- **Dependency injection:** services are injected via `Depends(get_xxxx_service)`. Services MUST NOT be constructed inline in controllers.

## 5. Cross-Repo Impact

- **Changing a BigQuery schema** requires coordinated updates in `data/` (DAG SQL), `traceability/` (ORM + API), and possibly `frontend/` (queries + components).
- **Adding a field to segments** touches: data DAG SQL, traceability ORM model + repository + service + controller, frontend query + component.
- **Any spec that changes shared data contracts** (BigQuery schemas, PostgreSQL tables used by both `data` and `traceability`) must explicitly document the cross-repo impact in the plan phase.

## 6. Code Quality

- **Formatter/Linter:** Ruff. Line length 120. Double quotes. 4-space indent.
- **Commits:** Conventional commits — `type(scope): description` (e.g., `feat(DEV-7058): deforestation override API`).
- **Tests:** pytest. Files named `test_*.py`. Auth mocked via `app.dependency_overrides[authorize]`. `PYTHONPATH=src` required. All tests use mocks — no test hits a real database.
- **Pre-commit hooks:** conventional-pre-commit, ruff, standard pre-commit-hooks.

## 7. Security

- **No secrets in code or specs.** No `.env` values, no service account keys, no tokens.
- **PII and sensitive data** must be encrypted at rest and handled per GDPR/compliance requirements.

## Amending the Constitution

These rules are immutable by default. To change one:

1. **Create a decision record** in the active spec's `decision-log.md`:
   - What rule is being changed
   - Why it no longer serves the project
   - What the new rule is
   - What existing code is affected

2. **Update this file** with the new rule. Add a comment with the date and reason:
   ```markdown
   - **New rule.** <!-- Amended YYYY-MM-DD: reason -->
   ```

3. **Update affected convention files** in `conventions/` to stay consistent.

Amendments should be rare. If you're amending frequently, the rules may be too specific.
